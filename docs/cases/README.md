# Как мы ведем текстовые кейсы

- Папка `docs/cases/` — сюда складываем текстовые постановки, чтобы было понятно, что нужно автоматизировать.
- Формат — обычный Markdown по шаблону ниже. Один файл на кейс или блок кейсов (как удобнее).
- После согласования я перевожу кейсы в кодовые тесты в `tests/...` и запускаю их.

## Шаблон
Смотри `docs/cases/TEMPLATE.md` — копируй и заполняй.

## Как добавлять
1. Создай новый файл в `docs/cases/`, например `auth-login.md` или `api-surveys.md`.
2. Заполни шаблон текстом (без кода).
3. Я беру файл, пишу тесты, обновляю статус кейса прямо в файле (например, добавляю блок «Статус: в работе/готов»).

## Минимальный пример
```
Название: [кратко, что проверяем]
Тип: UI | API
Приоритет: high | medium | low
Теги: выбери подходящее — например @ui @api @smoke @regression @auth @security @surveys @feedback @objectives @performance-review @org-structure @critical @high

Стенд: [номер/URL стенда]
Роль/данные:
- роль: [admin/user/manager/...]
- креды: [не вставляй логин/пароль; если не из .env, напиши откуда их взять]
- доп. данные: [id, payload, сущности]

Предпосылки:
- [что должно существовать до теста или что подготовить]

Шаги:
1) ...
2) ...
3) ...

ОР:
- [конкретные проверки: что должно быть видно/вернуться/измениться]

Нюансы:
- [skip при отсутствии данных? особые таймауты? очистка стейта? требования к окружению/браузеру?]

Статус:
- черновик | в работе | готов (заполняем, чтобы понимать прогресс)
```
